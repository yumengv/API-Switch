> Personal AI API Gateway & Management Center

## 1. 项目概述

**API Switch** 是一款基于 Tauri v2 的桌面应用，用于统一管理和转发多个 AI API 渠道。面向个人本地使用，默认信任本机环境，不按公网多用户服务进行安全模型设计。

### 产品定位

API-Switch：绿色便携的个人 AI 网关——多渠道路由、5 协议转换、智能容错，一站管理所有 AI 服务。

核心价值：
- **一个入口，告别零散笔记**：多家 AI 服务商的 API Key、模型、用量，集中管理、统一调用
- **多渠道路由 + 三级冷却熔断**：一个入口访问多家服务商，DB 冷却（模型级）、内存熔断器（渠道+模型级）、渠道冷冻（渠道级）三级兜底，故障自动 failover
- **5 套协议自由对接**：同时适配 OpenAI-compatible / OpenAI-Responses / Claude / Gemini / Azure 五套协议
- **全平台覆盖**：Windows x64 / macOS (x64+arm64) / Linux x64，有桌面用桌面，无头服务器用 Web Admin

## 2. 功能总览

### 已实现特性

#### API 服务配置
- 管理 OpenAI、Claude、Gemini、Azure、Custom 五类 API 渠道
- 渠道增删改查、URL 可达性探测、模型列表拉取
- 内置模型目录，为模型提供元数据信息

#### 模型管理与智能路由
- 模型管理页支持直接编辑模型别名，渠道与模型标识保持锁定，只允许调整展示别名
- 自定义模型分组，支持精确匹配 → 模糊匹配 → AUTO 组 fallback 的三级路由
- 三种排序策略：自定义顺序、最快优先、最新优先
- 冷却感知与熔断感知路由，故障自动 failover
- 空流检测：HTTP 200 OK 但 SSE 流 45s 内无有效数据视为失败，触发模型冷却

#### 代理协议兼容
- OpenAI 兼容本地入口，客户端直连即可使用
- 五套协议适配器，各自独立处理认证、请求/响应转换
- 支持 Claude / Gemini / Azure 到 OpenAI 格式的转换
- 支持 `/v1/responses` 兼容层

#### 故障转移与保护
- **三级容错体系**：DB 冷却（模型级，300-1800s）→ 内存熔断器（渠道+模型级，60s 窗口）→ 渠道冷冻（渠道级，6h），逐层兜底、互不阻塞
- 请求失败自动写日志、设置对应层级冷却、触发 failover
- **空流检测**：HTTP 200 OK 但 SSE 流 45s 内无有效输出判定为空流（DroppedEmpty），触发 L1 DB 冷却，避免上游"挂死不推"静默故障
- **非流式空响应检测**：HTTP 200 但响应体为空同样触发冷却
- 内存三态熔断器（Closed/Open/HalfOpen），连续 5 次失败跳开 60s，试探恢复
- 按状态码区分处理策略（401/403/410 保持单模型条目永久禁用，429/5xx 可恢复冷却）
- 支持按上游错误限定词触发渠道级 6 小时冷冻：同一渠道下所有模型临时退出路由，避免账号额度、密钥或上游通道级故障反复命中
- 流式请求结束原因记录（Done / Error / Dropped / DroppedEmpty）、敏感参数脱敏


#### 桌面端界面
- Dashboard、渠道管理、API 管理、令牌管理、使用日志、系统设置
- 实时统计卡片与图表、使用日志分页与筛选
- 测试对话弹窗、系统托盘菜单、欢迎引导、双语 UI

#### 设置与持久化
- SQLite 本地持久化，默认启用 WAL 模式
- 自动建表、字段升级兼容、默认配置补齐
- 代理端口、冷却阈值、默认排序、语言主题等可配置

#### 认证与安全
- 上游 API Key 与客户端 Access Key 分离
- Bearer Access Key 校验（可开关）
- Web Admin 登录、Token 校验、审计日志
- 定位为个人本地工具，外网部署需额外安全措施

#### Web Admin 与 CLI
- 同一套 React 页面适配桌面端和 Web Admin
- CLI 配置生成，支持 OpenCode、Claude Code、Codex CLI
- 检查 GitHub 最新版本、release 下载链接
- 多平台构建、Playwright E2E 测试

## 4. 设计取舍

| 项 | 当前取舍 | 原因 |
|---|---|---|
| API Key 明文存储 | 接受 | 个人本地工具降低复杂度 |
| Access Key 可关闭 | 接受 | 本机使用优先降低门槛 |
| SQLite + Mutex | 接受 | 单机低并发场景足够 |
| 冷却状态 DB 持久化 | 接受 | 重启后坏模型不会立即恢复 |
| CORS 宽松 | 接受 | 非公网服务 |
| Custom base_url 不拼接 /v1 | 接受 | 用户填写完整版本路径 |
| 不新增 API 类型处理中转站 | 接受 | 用 openai/custom 类型 + 手动添加模型即可 |

## 5. 开发路线图

### P0 — 协议架构调整

- **中立 IR + Capability Router**：建立自定义中立内部协议，以能力而非模型名路由上游

### P0 — Gemini 协议端点兼容

- **端点探测增强**：对不支持原生 Gemini 端点（/v1beta/openai/...）的服务端，需评估是否自动 fallback 或以其他方式尝试识别 Gemini 兼容性

### P0 — Web Admin 路线落地收尾

- **设置更新与版本冲突闭环**：完善 Web 端设置更新的版本一致性处理
- **登录 / token / 鉴权收尾**：修复 validateToken 网络异常时的逻辑缺陷

### P2 — Gemini 原生协议端点补全

- **countTokens / embedContent / batchEmbedContents**：直接转发 Gemini 上游的专用端点

### P2 — Claude 协议边角一致性优化

- **边角字段严格对齐**：补充 stop_sequence 等官数字段
- **thinking 输出面纯化**：明确官方语义与内部兼容承载的边界
- **流式 SSE 边角事件一致性补测**：补全极端事件组合测试

### P3 — 其他改进项

- **Tauri v2 安全基线**：收紧 CSP 策略、最小化 capabilities
- **单实例与窗口状态持久化**：避免多进程争抢、记住窗口位置
- **跨平台 WebView 兼容矩阵**：建立各平台冒线测试清单
- **全局错误语义统一**：统一 IPC 与 HTTP 错误结构、UI 反馈一致性

## 附录：核心价值

- **多渠道路由**：一个入口访问多个 AI 服务商，按模型自动匹配或手动指定
- **提升可用性**：模型冷却 + 自动故障转移，降低单渠道故障对使用的影响
- **用量可视化**：实时 Dashboard + 请求日志 + Token 消耗统计
- **轻量桌面**：Tauri v2 架构，内存占用低，跨平台便携
